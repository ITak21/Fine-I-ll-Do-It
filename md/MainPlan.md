# Main Plan

## 운영 규칙

1. 프로젝트 결정 사항과 계획은 `md` 디렉터리의 마크다운 문서로 기록한다.
2. 중간 계획 문서는 `md/MainPlan.md`로 유지한다.
3. 모든 assistant 응답은 `md/response(n).md` 형식으로 별도 기록한다.
4. `response(n).md`에는 요약을 쓰지 않는다.
5. `response(n).md`에는 assistant가 사용자에게 최종으로 보낸 응답 전문을 그대로 기록한다.
6. 모든 응답 전에 `ProjectDirectives.md`를 확인한다.
7. assistant는 한국어로 정중한 존댓말로 응답한다.
8. 모든 마크다운 파일은 UTF-8 인코딩으로 저장한다.

## 프로젝트 목표

1. 카드사별 엑셀 파일을 입력받아 사용 내역을 읽는다.
2. 사용 내역을 공통 거래 구조로 정규화한다.
3. 거래를 SQLite에 저장한다.
4. 카드사별 컬럼 매핑과 파서 차이를 흡수한다.
5. 월별 비교와 카테고리 분석이 가능한 구조를 만든다.
6. 결과는 로컬 PC에서만 동작하도록 유지한다.
7. 기술 스택은 `Tauri + React + TypeScript + SQLite`로 진행한다.

## 현재 완료 상태

1. Tauri + React + TypeScript + SQLite 기본 구조를 구성했다.
2. Windows `target` 잠금 문제를 우회하는 빌드 구성과 기본 아이콘 설정을 끝냈다.
3. SQLite 초기화, 스키마 생성, 기본 카테고리 시드를 적용했다.
4. 현대카드, 롯데카드, 신한카드, 하나카드 파서를 실제 파일 기준으로 보정했다.
5. 롯데카드는 실제 거래 시트와 헤더를 찾도록 보완했다.
6. 하나카드는 합계/여백 행을 제외하도록 수정했다.
7. 신한카드는 `총 n건` 합계 행이 저장되지 않도록 필터를 추가했다.
8. 상세 거래 목록 조회 SQL 오류를 수정해 최근 가져오기 결과가 정상 표시되도록 보완했다.
9. 가져오기 직후 상세 거래 목록이 바로 보이도록 연결했고, 목록 재조회가 실패해도 방금 파싱한 결과를 유지하도록 보완했다.
10. 가져오기 결과를 카드사 누적이 아닌 가장 최근 가져오기 1회분 기준으로 표시하도록 정리했다.
11. 승인시각을 기준으로 기존 거래와 비교해 중복 거래는 건너뛰고 새 거래만 저장하도록 중복 방지 로직을 추가했다.

## 카드사별 상태

1. 현대카드: 샘플 파일 기준 파싱 가능
2. 롯데카드: 실제 거래 시트 탐지 및 실제 헤더 기준 파서 반영 완료
3. 신한카드: 합계 행 제외 반영 완료
4. 하나카드: 합계/여백 행 제외 반영 완료

## 현재 남은 작업

1. 카드사별 파싱 결과가 실제 건수와 금액 기준으로 맞는지 계속 검증한다.
2. 카드사별 날짜 형식과 취소 금액 처리 규칙을 더 정밀하게 맞춘다.
3. 중복 판정 기준을 실제 카드사 데이터에 맞춰 추가 검증한다.
4. 카테고리 규칙 저장/적용 기능을 구현한다.
5. 월별 합계, 전월 비교, 카테고리 집계 화면을 구현한다.
6. 결과를 엑셀 파일로 내보내는 기능을 구현한다.
7. 가져오기 실패 사례와 예외 메시지를 계속 정리한다.

## 다음 우선순위

1. 카드사별 파싱 정확도와 중복 판정 기준 검증
2. 카테고리 규칙 설계 및 적용
3. 월별 분석/비교 화면 구현
4. 엑셀 내보내기 구현
5. 예외 케이스 및 오류 메시지 정리

## 메모

1. 현재는 로컬 개발 환경 기준으로 동작한다.
2. 빌드와 실행은 가능하지만 분석 UI와 내보내기 기능은 아직 미구현이다.
3. 일부 문자열 리소스와 UI 텍스트 정리는 계속 필요하다.
4. `response(n).md`는 반드시 최종 응답 전문 그대로만 기록한다.
